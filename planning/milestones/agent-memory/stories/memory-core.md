# Story: Core Memory Module + Amygdala Integration

> As a **researcher (Tyler)**, I want the agent to have persistent memory exposed as tools — `memory_recall` and `memory_remember` backed by a polymorphic `PersistentState` interface — so that the agent builds a coherent world model across sessions, memory is scoped and auditable like any other tool, and the architecture supports future backends (mmogit, embeddings) and interfaces (Reddit, CLI) without changes to consumers.

**Design doc**: `designs/memory-architecture.md` (produced in session 24, mem-01 through mem-05). This is the implementation spec.

## Acceptance Criteria

- `PersistentState` interface in `packages/llm/src/memory/` with `JsonFilePersistentState` implementation
- 4 capsule-shaped memory types (observation, learning, relationship, reflection) with Zod schemas — each carrying id, provenance, modality, uncertainty, visibility, tags, supersededBy, accessCount
- `createMemoryTools({ state })` factory returning `memory_recall` and `memory_remember` as `defineTool` instances
- Memory tools registered in tool registry; subagents get `memory_recall` only; `memory_remember` is orchestrator-mediated (Option B)
- route.ts calls `memory_recall` before amygdala, formats results as `memoryContext` string on `AmygdalaInput`
- route.ts writes memories post-interaction via `extractMemoryWrites`, gated by 4-band threat assessment (<0.3 full write, 0.3-0.49 write with elevated uncertainty, ≥0.5 no write)
- Memory trace events (`memory:recall`, `memory:write`) emitted through collector with shapes per design §4.6
- MemoryInspector frontend component (collapsible panel, entries by type with uncertainty badges, per-interaction recall/writes)
- Red-team: memory poisoning attempts gated by threat assessment (3 attack vectors)
- Integration test: cross-session continuity via shared `PersistentState`
- Persistence on Railway volume (`MEMORY_DATA_DIR` env var, defaults to `data/memory/`)
